# 写在前面
此实习工作主要是关于用excel采集公司年报，DM公司使用ai自动获取年报中的数据做成excel表格，但由于ai有时会读取错误，所以需要重新校验和修正。后续遇到具体科目时，优先按本 skill 中的“科目判断 + 表格口径 + 核查字段 + 易错点”来执行。

# 一、DM实习总目标
- 核查 AI 从年报 / 审计报告中抽取出的 Excel 数据是否正确
- 发现错误时，以原始 PDF / 年报附注为准进行修正
- 尽量沉淀可复用的判断规则，而不是只处理单个表

# 二、通用工作流
1. 先确认公司、报告期、PDF 链接、page_id 是否正确
2. 定位具体科目属于哪一类往来科目
3. 判断当前采集表属于哪种统计口径
4. 对照原始附注逐字段核查
5. 发现错误立即修正，并保留清晰的核查依据
6. 如果遇到稳定高频错误，优先沉淀为 skill 规则

# 三、往来科目分类规则

## 1. 资产类：别人欠我 / 我先付出去
### 应收账款（经营性应收）
- 属性：流动资产
- 核心定义：主营业务销售商品、提供劳务形成的应收款
- 典型场景：卖货赊销、工程服务未回款、主营业务欠款
- 附注重点：账龄、坏账准备、前五大欠款方名称/金额/占比

### 其他应收款（非经营性应收）
- 属性：流动资产
- 核心定义：不是主营业务产生的应收、暂付款项，属于杂项应收
- 典型场景：员工备用金、借款、关联方拆借、押金、保证金、应收赔款、罚款、政府补助待收
- 附注重点：款项性质分类、坏账准备、账龄、大额对手方明细

### 预付账款
- 属性：流动资产
- 核心定义：预先付给供应商的钱，但货 / 服务还未到位
- 典型场景：材料款、设备款、工程款、外包服务费预付
- 附注重点：款项性质拆分、账龄结构、大额预付对象、关联方预付

## 2. 负债类：我欠别人 / 别人先付给我
### 应付账款（经营性应付）
- 属性：流动负债
- 核心定义：主营业务采购物资、接受服务形成的应付款
- 典型场景：采购原材料赊账、服务费未付
- 附注重点：性质列示、账龄分布、大额逾期应付、无能力支付款项说明

### 其他应付款（非经营性应付）
- 属性：流动负债
- 核心定义：非主营业务形成的应付、暂收款项，属于杂项应付
- 典型场景：押金、保证金、关联方借款、股东借款、暂收代付款、预提费用、滞纳金、罚款、员工报销未付
- 附注重点：款项性质拆分、关联方往来余额、拆借利率、还款安排

### 预收账款
- 属性：流动负债
- 核心定义：提前收到客户货款 / 服务费，但还未履约
- 典型场景：预收货款、会员预存费、项目预付款
- 新准则注意：很多企业把主营业务预收款计入“合同负债”，不能机械认为所有预收都在“预收账款”
- 附注重点：期末预收款对应项目、履约进度、期后结转收入金额

# 四、统一采集口径
这几类往来科目通常按以下 4 个口径采集：
- 1：分类性质
- 2：账龄
- 3：超一年
- 4：重要 / 前五大

## 1. 分类性质表
关键字段：
- `unit`
- `report_date`
- `statistics_caliber`：必须为“1分类性质”
- `category_code`
- `original_item_name`
- `ending_balance`

核查重点：
- 报告期是否为目标日期（如 2025/12/31）
- original_item_name 是否有错别字
- ending_balance 是否与原表一致
- 统计口径字段位置可能混乱，要以实际含义判断，不要只看列位置

## 2. 账龄表
关键字段：
- `unit`
- `report_date`
- `statistics_caliber`：必须为“2账龄”
- `category_code`
- `original_item_name`：此处一般应为账龄区间
- `ending_balance`

核查重点：
- original_item_name 在这里通常不是公司名，而是账龄期间
- 账龄区间不要抄错，如“1年以内”“1-2年”“2-3年”等

## 3. 重要明细表
关键字段：
- `unit`
- `report_date`
- `statistics_caliber`
- `category_code`
- `original_item_name`：一般为具体公司名
- `ending_balance`
- `aging`
- `payment_nature_desc`
- `reason`
- `relation_description`

口径判断：
- 标题中含“超1年” → 用口径 3
- 标题中含“重要”“前五大”“期末余额重要” → 用口径 4

核查重点：
- 公司名、金额、账龄、原因、关联关系要和附注逐项对应
- aging / relation_description 等字段若报告未披露，可允许为空，但不能乱填

# 五、易错点
## 1. 经营性 vs 非经营性不能混
- 应收账款 / 应付账款 = 主营业务、经营性
- 其他应收款 / 其他应付款 = 非主营业务、杂项往来

## 2. 其他应收、其他应付的分类性质表最容易采错
- 不能看见表就采，要确认是不是目标子项对应的那一张
- 特别是“其他应付款”下可能有多个表，一般优先采“其中：其他应付款”对应的那张

## 3. 预收账款要注意新收入准则
- 很多企业会把主营业务预收款放入“合同负债”
- 看到预收账款时，要结合附注口径和标题判断

## 4. 不要机械信任 AI 抽取结果
- 金额、公司名、账龄、口径、字段含义都要回到原始 PDF 复核
- 有歧义时，以附注标题、上下文和披露结构为准

# 六、年报 Excel / PDF 核对标准流程
这部分吸收自 Claude Code 同步过来的年报核对经验，后续遇到政府补助、其他收益、往来科目等表格核对时优先复用。

## 1. 定位原始 PDF 表格
- 优先查看 Excel 中的 `page_id`
- 目标表格通常在标注页附近，常见范围是当前页前后 1-2 页
- 如果表格跨页，继续检查后续页面，不要只看单页

## 2. 提取与识别策略
### 基础人工核对策略
- 先看附注标题，确认当前表就是目标表
- 不要只凭金额搜索，先确认表名、科目名、上下文
- 注意单位（通常是“元”），避免误判数量级

### 自动化策略（适合批量核对）
- 主方案：**Camelot** 本地提取 PDF 表格
- 备用方案：**百度 OCR** 识别复杂表格
- 优先逻辑：
  1. 先用 Camelot
  2. 若提取结果过少或结构异常，再考虑 OCR
- 多页搜索：
  - Camelot 可搜索当前页及后续多页
  - OCR 作为备用，控制成本使用

## 3. 表格过滤规则
这是高优先级规则。
- PDF 中往往不止一个表，不能把当前页所有表都当目标表
- 应先判断表格前几行是否包含目标关键词
- 只处理包含目标标题 / 目标科目的表
- 合计 / 小计行要谨慎，避免误匹配成明细行

## 4. 逐行核对原则
- 必须逐行比对，不只核对合计
- 每一行至少核对：
  - 项目名称
  - 本期金额 / 期末余额
  - 上期金额 / 期初余额
  - 必要时核对关系字段、账龄、性质说明
- 空值、横杠、nan 需要按原表语义理解，不能机械判错
- 允许极小的小数精度差异，但不能放过数量级错误

## 5. 常见错误模式
- 多一个 0 / 少一个 0
- 小数点位数异常
- 本期与上期互换
- 合计行被误采成明细行
- 采错表（尤其其他应收 / 其他应付的分类性质表）
- 采到了同页其他科目表格的数据

## 6. 发现错误后的处理原则
- 立即以 PDF 原文为准修正 Excel
- 明确记录：错在哪、依据是什么、应改成什么
- 如果任务场景允许，可对修改行做醒目标记（如整行标黄）
- 同类错误重复出现时，优先回头补规则，而不是只改单条数据

## 7. 批量任务中的工具选择经验
- PyPDF2：文本层面可用，但表格结构丢失严重，不适合复杂表格核对
- pdfplumber：比纯文本提取更好，但复杂格式仍可能不稳
- Camelot：适合本地批量 PDF 表格提取，是主力方案
- OCR：适合作为 Camelot 失败时的兜底方案

## 8. 执行中的优先级
- 原始 PDF / 年报附注 > AI 抽取结果 > 现有 Excel
- 表名和附注上下文 > 单个数字片段
- 可复用规则 > 一次性修补

## 9. 应付账款分类场景的补充实操规则（2026-04-30）
### 起手式
- 若 Excel 存在 `_memo` 列且其中直接给出 PDF 链接，优先先把涉及 PDF 下载下来，再开始逐家公司核对
- 再结合 `page_id` 到对应页前后 1-2 页定位目标表

### 人工核查顺序
- 先按公司分组，再判断该公司当前行属于哪一类：
  - 分类性质表
  - 超 1 年重要应付账款
  - 是否存在逾期尚未支付中小企业款项
  - 标题被上下文拼接污染的变体表
- 不要把整份“应付账款分类”Excel 机械当成同一种口径

### 修改与标记规则
- 只在与 PDF 原文明确不一致时修改 Excel
- **只对实际发生数值改动的单元格标黄**
- 如果只是核实无误，没有改值，则不标黄
- 修改后必须执行：**改值 → 保存 → 重开 → 验证值和颜色是否真正写入**

# 七、执行原则
- 先判断科目，再判断口径，再逐字段核查
- 有截图 / page_id / PDF 原文时，以原始材料为最高优先级
- 对重复出现的错误，优先沉淀成规则
- 回答用户时，尽量明确指出：错在哪里、依据是什么、应该改成什么
