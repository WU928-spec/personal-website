### **任务 3: 类型安全改进**

**问题：**
- Home.tsx 中有 1 个 `any` 类型
- API 响应没有运行时验证

**解决方案：**
```typescript
// 创建 src/types/api.ts
import { z } from 'zod'

// GitHub API 响应验证
export const GitHubRepoSchema = z.object({
  name: z.string(),
  description: z.string().nullable(),
  language: z.string().nullable(),
  stargazers_count: z.number(),
  forks_count: z.number(),
  updated_at: z.string(),
})

export type GitHubAPIRepo = z.infer<typeof GitHubRepoSchema>

// 安全的解析函数
export function parseGitHubRepos(data: unknown): GitHubAPIRepo[] {
  try {
    return z.array(GitHubRepoSchema).parse(data)
  } catch (error) {
    console.error('GitHub repos validation failed:', error)
    return []
  }
}
```

**应用：**
```typescript
// Home.tsx
const validatedRepos = parseGitHubRepos(data)
const repos: GitHubRepo[] = validatedRepos.map((r: GitHubAPIRepo) => ({
  // 现在有完整的类型提示
}))
```

**成果：**
- ✅ 消除所有 `any` 类型
- ✅ 编译时类型检查
- ✅ 运行时数据验证
- ✅ 自动错误处理

## 📚 学到的经验

### **1. 重构前先规划**
- 明确目标和优先级
- 分阶段执行，每个阶段都有明确产出
- 避免一次改动太多

### **2. 保持向后兼容**
- 合并 Context 时保留旧的 hooks
- 自动迁移旧数据格式
- 渐进式重构，不破坏现有功能

### **3. 测试驱动重构**
- 先写测试，再重构
- 测试通过才算完成
- 防止引入新 bug

### **4. 类型安全很重要**
- 编译时类型检查（TypeScript）
- 运行时数据验证（Zod）
- 双重保障，更加安全

### **5. 性能优化要有数据支撑**
- 使用 React DevTools Profiler
- 只优化真正的瓶颈
- 不要过早优化
